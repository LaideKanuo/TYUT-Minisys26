## 三、关键技术集成错误（4项，当前代码无法编译）

### 集成错误 #5 — minisys.v 新控制器端口全部被注释，导致关键信号悬空

| 项目 | 内容 |
|------|------|
| **严重程度** | 致命 |
| **问题位置** | minisys.v 第 128-144 行 control32 实例化 |
| **错误代码** | 第 122 行 `//.MemorIOtoReg(MemorIOtoReg)` — 被注释掉；第 134 行 `//.MemorIOtoReg(MemorIOtoReg)` — 被注释掉 |
| **问题分析** | control32.v 模块定义中**已经增加了** 5 个新端口：`Alu_resultHigh`（输入）, `MemorIOtoReg`, `MemRead`, `IORead`, `IOWrite`（输出）。但 minisys.v 中这 5 个端口的连接**全部用 `//` 注释掉了**。Verilog 中未连接的模块端口**不是 optional**——如果模块端口在定义中存在但实例化时未连接，综合工具会报错或产生隐式连接（`wire` 自动生成但值永远为 Z） |
| **实际后果** | ① `Alu_resultHigh` 输入悬空 → control32 无法判断当前地址是内存还是 IO（`is_io` 永远为不确定值 X）→ `IORead`/`IOWrite`/`MemRead`/`MemWrite` 输出全错 ② `IORead`/`IOWrite` 输出悬空 → memorio 的 `ioread`/`iowrite` 输入永远为 Z → 所有 IO 操作失效 ③ `MemRead` 输出悬空 → 内存读操作失效 |
| **修正方向** | 取消所有 `//` 注释，正确连接即可；同时需要在 minisys.v 顶部补充 `wire ioread, iowrite, memread;` 声明（目前实验二的 minisys.v 已声明了 `ioread`/`iowrite`/`memread` 这三个 wire，只需取消注释端口连接） |

### 集成错误 #6 — idecode32.v 的 MemtoReg/MemorIOtoReg 端口悬空

| 项目 | 内容 |
|------|------|
| **严重程度** | 致命 |
| **问题位置** | minisys.v 第 112-126 行 idecode 实例化 |
| **错误代码** | 第 122 行 `//.MemorIOtoReg(MemorIOtoReg)` 被注释掉 |
| **问题分析** | idecode32.v 第 53-60 行是写回数据三路 MUX：`Jal → opcplus4 / MemtoReg → read_data / default → ALU_result`。idecode 模块期望连接到 `MemtoReg` 端口 → 实验二改名为 `MemorIOtoReg` → 这个端口在 minisys.v 中被注释 → **写回数据选择逻辑的第二路（LW 的数据回写）永远无法触发** |
| **后果** | 所有 LW 指令（包括从 IO 读数据）都无法正确写回寄存器，CPU 访存功能完全异常 |

### 集成错误 #7 — control32.v 内部 is_io 信号重复定义（编译错误）

| 项目 | 内容 |
|------|------|
| **严重程度** | 高 |
| **问题位置** | control32.v 第 37 行和第 85 行 |
| **错误代码** | 第 37 行 `wire is_io = (Alu_resultHigh[21:0] == 22'b1111111111111111111111);` 第 85 行 `wire is_io_space = (Alu_resultHigh[21:0] == 22'b1111111111111111111111);` |
| **问题分析** | 两者逻辑完全相同，但第 37 行的 `is_io` 变量在第 87-91 行的 `MemRead`/`IORead`/`MemWrite`/`IOWrite` assign 中引用。第 85 行的 `is_io_space` 定义后**从未被使用**。同时 `is_io` 在第 37 行用 `assign wire` 语法定义了一次，又在别处被引用——这不构成 Verilog 语法错误（重复定义的话才会报错），但它是一个未连接的重复变量。 |
| **具体判定**：第 37 行定义了 `wire is_io` 且被实际使用（第 87-91 行）。第 85 行定义了 `wire is_io_space` 但未被使用——Vivado 会报 warning（未使用信号），但不是 error。实际上这不会导致编译失败，但说明**代码是碎片化的、未清理的草稿状态**。 |
| **修正方向** | 删除第 37 行的 `wire is_io`（它和第 85 行的 `is_io_space` 重复），统一使用 `is_io_space`，或反过来。关键是在 minisys.v 中正确连接 `Alu_resultHigh` 输入（见 #5），否则 `is_io` 永远是 X |

## 四、逻辑错误（2项，不独立构成编译错误，但功能不正确）

### 逻辑错误 #9 — JR 指令 RegWrite 排除逻辑不完全可靠

| 项目 | 内容 |
|------|------|
| **严重程度** | 中 |
| **问题位置** | control32.v 第 79 行 |
| **当前代码** | `assign RegWrite = (R_format || Lw || Jalc || I_data) && !(Jrn);` |
| **判定** | 这里的 `!(Jrn)` **在逻辑上是正确的**——JR（R_format=1, Function=001000）时 Jrn=1，RegWrite 被置 0。这实际修复了实验一的 Bug #8。但请注意依赖此修复的前提是 Alu_resultHigh 正确接入（#5），否则整体控制器输出全错 |

### 逻辑错误 #10 — JAL 指令写回目标寄存器 Bug 未修复（继承自实验一 Bug #7）

| 项目 | 内容 |
|------|------|
| **严重程度** | 高（潜伏 Bug，在特定条件下会暴露） |
| **问题位置** | idecode32.v 第 46-51 行 |
| **错误代码** | `always @* begin if (RegDst == 1'b1) write_register_address = write_register_address_1; else write_register_address = write_register_address_0; end` |
| **问题分析** | 与实验一完全相同：JAL 指令(opcode=000011) `RegDst=0`（非 R 型），走 else 分支选 `rt` 字段 → `instruction[20:16]`。但 JAL 指令的 bit[20:16] 是**跳转地址的一部分**，不是寄存器号。MIPS 规范规定 JAL 隐式写入 `$31`（`$ra`）。这会在跳转目标地址的 bit[20:16] ≠ 00000 时随机破坏一个寄存器 |
| **为什么现在没暴露** | 凑巧：目前的测试程序中 JAL 目标地址的 bit[20:16] = 00000 → 选了 `$0` → `$0` 写保护拦截 → 静默丢弃。换一个测试程序立即暴露 |
| **修正** | 在 `write_register_address` 的 always 块中优先判断 `Jal`：`if (Jal) write_register_address = 5'b11111; else if (RegDst) ...` |

## 五、实验一二关键接口不匹配（3项）

### 不匹配 #11 — 控制器端口名变更导致的级联重连需求

| 实验一 | 实验二 | 影响范围 |
|--------|--------|---------|
| `control32.v` → 输出 `MemtoReg` | `control32.v` → 输出 `MemorIOtoReg` | minisys.v 中 idecode 的 `.MemtoReg(memreg)` 需改为 `.MemorIOtoReg(memoriotoreg)` 或兼容对接 |
| control32 无 Alu_resultHigh 输入 | 增加 `Alu_resultHigh[21:0]` 输入 | minisys.v 需增加 `assign alu_result_high = alu_result[31:10];` 并连接到 control32 |
| control32 无 IORead/IOWrite/MemRead 输出 | 增加 `IORead`/`IOWrite`/`MemRead` 输出 | minisys.v 中需连接到 memorio 和 ioread 的对应端口 |

### 不匹配 #12 — idecode32.v 端口名未同步

idecode32.v 的端口 `MemtoReg`（第 14 行）在实验一中是 `MemtoReg`，实验二中被改为连接 `MemorIOtoReg`。但 idecode32.v 内部逻辑（写回数据三路选择第 56 行 `else if (MemtoReg == 1'b1)`）仍然使用端口名 `MemtoReg`。问题在于 minisys.v 第 122 行 `.MemorIOtoReg(...)` 试图驱动一个名为 `.MemorIOtoReg` 的端口，但 idecode32.v 的端口名叫 `MemtoReg`。

**这在 Verilog 中不会报错**——因为端口连接是"按名称关联"的，错误会被静默接受，但 idecode32 的 `MemtoReg` 输入实际由什么信号驱动取决于 minisys.v 中 `memreg` wire 连到了哪里。当前代码第 122 行被注释掉了，所以是**未连接**（见 #6）。

### 不匹配 #13 — 数码管模块缺失导致顶层端口定义不完整

minisys.v 顶层模块端口只有 `fpga_rst, fpga_clk, switch2N4[23:0], led2N4[23:0], start_pg, rx, tx`。如果数码管需要连接到 FPGA 外部引脚，需要在 minisys.v 顶层增加数码管输出端口（如 `seg[7:0]`, `an[3:0]` 等）。当前完全没有。
