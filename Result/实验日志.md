#### 工作内容

**1. 取指单元（ifetc32.v）实现**

将原模板中 `PC_plus_4[31:2] = ������` 等占位符替换为实际逻辑。

- **PC+4 计算**：32 位指令 4 字节对齐，PC+4 = 高 30 位加 1、低 2 位恒为 0。`PC_plus_4[31:2] = PC[31:2] + 1'b1; PC_plus_4[1:0] = 2'b00`
- **next_PC 优先级链**：JR > BEQ(Zero=1) > BNE(Zero=0) > J > JAL > 顺序执行(PC+4)。
  J 和 JAL 跳转地址计算公式相同：`{PC_plus_4[31:28], Instruction[25:0], 2'b00}`，区别仅在于 JAL 额外保存 `PC+4` 到 `$ra` 寄存器
- **PC 寄存器**：下降沿更新（与上升沿写寄存器的 idecode32 错开，避免同周期冲突）。reset=1 时 PC 和 opcplus4 清零

**2. 译码单元（idecode32.v）实现**

- **指令字段提取**：rs[25:21]、rt[20:16]、rd[15:11]、immediate[15:0]、opcode[31:26]
- **寄存器堆**：32×32bit，`$0` 恒为 0（写使能时 `write_register_address != 5'b00000` 保护）
- **立即数符号扩展** 16→32 位：`{16{instruction[15]}, instruction[15:0]}`
- **目标寄存器选择**（MUX）：RegDst=1 → rd（R型），RegDst=0 → rt（I型）
- **写回数据选择**（三路 MUX）：JAL→opcplus4 / MemtoReg→read_data / 其他→ALU_result

**3. 控制器（control32.v）实现**

补全原模板中 11 个空白的 assign，生成全部 13 个控制信号：

| 信号 | 为 1 的条件 | 作用 |
|------|-----------|------|
| RegDST | R_format (opcode=000000) | 目标寄存器选 rd 而非 rt |
| ALUSrc | I_format(001xxx) 或 LW(100011) 或 SW(101011) | ALU 第二操作数选立即数 |
| MemtoReg | Lw (100011) | 写回数据来自内存 |
| RegWrite | R_format 或 I_format 或 Lw 或 Jal | 写使能 |
| MemWrite | Sw (101011) | 内存写使能 |
| Branch | BEQ (000100) | 相等分支 |
| nBranch | BNE (000101) | 不等分支 |
| Jmp | J (000010) | 无条件跳转 |
| Jal | JAL (000011) | 跳转并链接 |
| Jrn | R_format 且 Function=001000 | 寄存器跳转 |
| I_format | opcode[5:3]=001 | 纯 I 型 ALU 指令标记 |
| Sftmd | R_format 且 Function[5:3]=000 | 移位指令标记 |
| ALUOp[1:0] | bit1=R或I型, bit0=BEQ或BNE | ALU 控制编码高位 |

**4. 执行单元（executs32.v）实现**

- **ALU 运算**（8 种，3 位控制码）：AND(000) / OR(001) / ADD(010) / ADDU(011) / XOR(100) / NOR(101) / SUB(110) / SUBU(111)
- **6 种移位指令**：固定移位 SLL/SRL/SRA（按 shamt 字段），可变移位 SLLV/SRLV/SRAV（按 rs 寄存器值）
- **SRA 算术右移**用 `$signed(Binput) >>> Shamt`，高位填符号位；SRL 逻辑右移用 `Binput >> Shamt`，高位填 0
- **Zero 标志**：`(ALU_output_mux == 0)`
- **分支地址**：`Add_Result = PC+4 + (Sign_extend << 2)`

**5. 外设模块实现**

- **leds.v**：24 位 LED 输出，分两次写入（ledaddr[1:0]=00→低16位, =01→高8位），上升沿 + 片选 + 写使能

**6. 顶层模块（minisys.v）连线**

6 个模块实例化共约 59 处 `???` 占位符逐个替换为正确的 wire 名。

**问题 #7 — JAL 指令写回目标寄存器地址错误（严重潜伏 Bug）**

| 项目 | 内容 |
|------|------|
| **来源文件** | idecode32.v 第 47-50 行（我们写的） |
| **错误代码** | `write_register_address = RegDst ? rd : rt` |
| **问题分析** | JAL 指令编码为 `|000011|instr_index[25:0]|`，**既没有 rd 字段也没有 rt 字段**。MIPS 架构规定 JAL 隐式写入 `$31`（即 `$ra`，返回地址寄存器）。但代码中 `RegDst=0`（JAL 不属于 R 型），导致选了 `instruction[20:16]`（跳转目标地址中的 5 个 bit）作为目标寄存器号——这是一个**随机值** |
| **为什么没暴露** | 测试程序中 JAL 的跳转目标地址较小，其 bit[20:16] 恰好全是 0 → 选中的是 `$0` → `$0` 写保护（idecode32.v 第 67 行 `write_register_address != 5'b00000`）→ 写入被丢弃。仿真波形中看不到任何异常！ |
| **正确写法** | `if (Jal == 1'b1) write_register_address = 5'b11111; // 显式写 $31`

---

**问题 #8 — JR 指令 RegWrite 未排除（潜伏 Bug）**

| 项目 | 内容 |
|------|------|
| **来源文件** | control32.v 第 32 行（我们写的） |
| **错误代码** | `assign RegWrite = (R_format \|\| I_format \|\| Lw \|\| Jal) ? 1'b1 : 1'b0;` |
| **问题分析** | JR 指令 opcode=000000 → `R_format=1` → `RegWrite=1`。但 **JR 指令只做寄存器跳转，不应该写目标寄存器**。JR 编码中 rd 字段恰好是 00000，所以写入目标为 `$0`，被写保护拦截了 |
| **为什么没暴露** | 同上，凑巧写入 `$0`，写保护生效 |
| **正确写法** | `assign RegWrite = ((R_format && !Jrn) \|\| I_format \|\| Lw \|\| Jal) ? 1'b1 : 1'b0;` |

#### 学习与发现

1. **MIPS 指令编码的特殊情况**：JAL 隐式写 `$31`，JR 不写寄存器——这两条指令打破了"R 型必写 rd"和"跳转指令不写寄存器"的常规模式，是编码时需要格外注意的边界情况。**仿真通过 ≠ 代码正确**——#7 和 #8 两个 Bug 都在仿真中完美"掩盖"了，靠人工代码审查才能发现。

2. **"凑巧不暴露"的 Bug 最危险**：如果测试程序的 JAL 跳转地址的 bit[20:16] 不为 00000（这在正常程序中完全可能），就会写入一个随机寄存器，数据被悄悄破坏，极难排查。

3. **Vivado 2019.x 的路径和编码坑**：中文路径不兼容（.coe 加载失败）、GBK/UTF-8 编码不一致（注释乱码）、IP 命名不一致（uart_bmpg_0 vs upg）——这些都不是逻辑错误但会严重阻碍进度。

4. **FPGA 芯片封装与引脚约束严格对应**：CSG324（324脚 Chip-Scale BGA）和 FGG484（484脚 Fine-Pitch BGA）虽然都是 Artix-7，但 Ball Grid Array 物理排布完全不同。约束文件中的每个引脚名必须存在于目标封装的引脚列表中。

5. **PLL/MMCM 的仿真行为**：锁相环不是组合逻辑，它需要物理锁定时间。仿真中 PLL 在锁定前输出为 0，这在纯数字逻辑仿真中容易被忽视。仿真时间要覆盖 PLL 锁定 + reset 释放 + 足够多的指令周期。

6. **BUFG 的布线限制**：全局时钟缓冲器 BUFG 有专用时钟走线要求。当非时钟信号（如按键）通过 BUFG 时需要 `CLOCK_DEDICATED_ROUTE FALSE` 放宽布线约束。

7. **组合逻辑中自赋值的 latch 推断**：FPGA 中组合逻辑必须完备——每个分支都必须给输出赋值。`a = a` 在硬件中等价于"保持上一值"，综合工具只能用 latch 实现。正确做法是用 `else a = 0` 或在块首赋默认值。

8. **模板代码的 Bug 不可轻信**：学生资料包是东南大学教学用模板，但 minisys_sim.v 的重复 endmodule、xdc 的注释掉的约束——说明即使是"官方"发布的模板也需要仔细审查，不能拿来就用。

#### 学习与发现

1. **地址译码原理**：通过 `Alu_resultHigh[21:0]`（ALU 结果的高 22 位）判断访问目标。IO 地址空间 `0xFFFFFCxx` 的高 22 位恒等于 `0x3FFFFF`，RAM 地址的高 22 位为 `0x000000`，通过比较即可区分。这就是 Memory-Mapped I/O 的硬件实现基础。
